# MetaSignal BI Layer — Executive Experimentation & KPI Governance Dashboard

A Tableau-based executive reporting layer built on top of the MetaSignal experimentation
platform. Translates MetaSignal's statistical audit outputs into decision-ready dashboards
for Product, Growth, and Engineering leadership.

---

## Overview

MetaSignal runs the statistical validity checks for every experiment: SRM detection,
peeking risk, guardrail monitoring, practical significance, pre-period balance. The BI
layer answers the next question every stakeholder asks: **"So what do we do?"**

Five dashboard tabs, each designed around a specific decision and audience:

| Tab | Audience | Decision Enabled |
|-----|----------|-----------------|
| Executive Overview | VP / Director | Is our experimentation program healthy? |
| Experiment Readout | DS / Product Manager | Should we ship this experiment? |
| Guardrail Monitoring | Engineering / Platform | Is anything breaking? |
| Metric Quality Panel | Data Science | Can we trust the numbers? |
| Segment Deep Dive | DS / Growth Analyst | Which segments drive the lift? |

## Data

All data is generated by `generate_bi_extracts.py`, which is structurally faithful to
MetaSignal's actual audit output schema. Data is synthetic but realistic:

- 25 experiments across 6 teams (Growth, Payments, Onboarding, Retention, Search, Monetisation)
- 2 SRM-failed experiments (EXP-1004, EXP-1017) — invalidated, not available for segment analysis
- 10 shipped, 10 held/in-review, 3 rejected, 2 shipped-with-note
- 11 guardrail warnings across 4 guardrail metrics
- Realistic lift distribution: null effects, small lifts, meaningful lifts, negative results

Regenerate data at any time:
```bash
python bi_tableau/generate_bi_extracts.py
```

## SQL Layer

The `sql/` directory documents the analytical logic between raw CSVs and dashboard views.
In this implementation, Tableau connects directly to CSVs and the SQL files serve as:
- Authoritative metric definitions (what each field means and how it's computed)
- The specification for any future database implementation
- Code-reviewable documentation of every derived field

## Design Principles

**Decisions over displays.** Every tab is built around a specific decision, not a
display of available data. The guardrail tab answers "is anything breaking?" — not
"here are all the guardrail numbers."

**FAIL reserved for structural impossibilities.** SRM is the only FAIL-level finding
because it's the only condition with no legitimate explanation. Everything else
produces WARN, enabling contextual judgment.

**Color used with meaning.** Green = pass/positive/healthy. Amber = warn/marginal.
Red = fail/negative/critical. No default Tableau blue for status encoding.

**Documented, not assumed.** Every metric definition, threshold, design decision,
and QA step is written down. See `docs/`.

## How This Connects

This BI layer is one component of a larger quality infrastructure:

- **MetaSignal** (this repo) — experiment validity platform + BI reporting layer
- **TrialCheck** — single-experiment readout auditor, zero dependencies, PyPI-published
- **MetricLens** — KPI definition and metric governance framework
- **RiskFrame** — end-to-end credit risk ML system with monitoring and decision layer

The experimentation → BI → metric governance → risk decisioning chain covers the
full analytical stack from hypothesis design to executive reporting.
